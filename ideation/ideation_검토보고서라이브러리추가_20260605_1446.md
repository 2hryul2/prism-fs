# Ideation — 검토보고서(분기연결검토보고서) 라이브러리 1급 노출 + 비교·질의 지원

- 작성: 2026-06-05 14:46
- 요청: 라이브러리에서 분기보고서(report)뿐 아니라 **분기검토보고서(review)** 를 선택 대상으로 추가하고, 비교조회·자연어질의(RAG)를 검토보고서로도 수행
- 결정: ① 검토보고서 pill **추가**(분기·검토 둘 다 유지) ② **전체 탭 확장** 적용

---

## 1. 배경 / 문제 정의

- 현재 라이브러리 셀의 `연결`/`별도` pill은 **모두 분기보고서(report.pdf)** 안의 주석 그룹이다.
- 사용자 요구: 외부감사인이 검토한 **분기연결검토보고서(review.pdf)** 의 연결 주석을 별도 대상으로 선택·비교·질의해야 함.
- 검토보고서 데이터·인덱스(`index_review.json`)는 **이미 수집·인덱싱 완료**(연결 30건, 별도 0건, source_type=slim).

## 4. 제약사항 / 극복

- **제약**: 선택키가 `회사/기간#fs_div` 2축뿐 → 문서축(doc_type) 부재.
  - **극복**: 키를 `회사/기간#doc_type#fs_div`(예 `신한/2026Q1#review#연결`)로 확장. parseSelKey/selKey 동시 수정.
- **제약**: 검토보고서는 별도 주석이 없음(slim).
  - **극복**: review pill은 연결만 렌더. 별도 요청 시 빈 결과 → "검토보고서는 연결만 제공" 안내(기존 stale "별도 미추출" 문구는 분기보고서엔 부정확하므로 함께 정리).
- **제약**: PDF 뷰어가 report.pdf 고정(index.html:1067·1071).
  - **극복**: `renderPdfRange(…, doc_type)` 인자 추가 → review 선택 시 `/api/pdf?doc_type=review`.

## 6. 수정 범위 (확정)

### 백엔드 (app.py) — doc_type 파라미터화
- `_load_index(…, "report")` 하드코딩 5곳에 `doc_type` 전파:
  - rag(1887), account-refs(1781), topic-map(1812), compare-memo(1838), terms/suggest(1968)
- 각 라우트 시그니처에 `doc_type: Literal["report","review"]="report"` 추가(기존 호출 하위호환).
- `/api/compare`, `/api/coverage` 는 payload의 doc_type 사용(이미 지원) — 검증만.

### 프론트엔드 (index.html)
1. 선택키 3축 확장: `selKey`/`parseSelKey`/`SELECTED`
2. 라이브러리 셀: 검토보고서(연결) pill 추가 — `review_indexed && review_notes_count_연결>0`
3. 비교조회: targets 매핑에 doc_type 반영(이미 백엔드 지원)
4. RAG 탭: 문서 소스 토글(분기/검토) → doc_type 전달
5. 커버리지/주제맵/XBRL: 문서축 선택자(INSIGHT_DOCTYPE 등) + doc_type 전달
6. `renderPdfRange` doc_type 인자화 → review.pdf 로드
7. 패널/결과 라벨: "📄 분기보고서·연결" vs "📋 검토보고서·연결" 구분
8. stale "별도 미추출" 경고문(1018·1365) 현행화
